# Design Document: Comprehensive Testing of `src/lib/`

**Date:** 2026-03-31

## Goal
Provide full test coverage for every function under `src/lib/` to ensure reliability, catch regressions, and support future development.

## Scope
All TypeScript files in `src/lib/` will receive unit tests, aiming for ≥ 90 % statement coverage.

## Testing Strategy
1. **Jest Unit Tests** – One test file per source file (`*.test.ts`). Tests will:
   - Verify happy‑path behavior.
   - Exercise error handling and edge cases.
   - Mock external dependencies (S3, BullMQ, filesystem) using `jest.mock`.
2. **Property‑Based Tests** (optional for pure utilities) – Use `fast-check` to generate random valid inputs for functions like parsers and diff generators, asserting invariants.
3. **Coverage Enforcement** – Add a CI step that runs `npm test -- --coverage` and fails if overall coverage drops below 90 %.

## Test Organization
```
src/__tests__/
  lib/
    vuln/
      vulnLoader.test.ts
      scanner.test.ts
    sbom/
      parser.test.ts
      generator.test.ts
    diffReports.test.ts
    utils.test.ts
    ...
```
Each test file mirrors its source counterpart, making navigation straightforward.

## Mocking Strategy
- **S3 / storage** – Mock the `@aws-sdk/client-s3` client to return predictable data.
- **BullMQ / job queues** – Mock queue methods (`add`, `process`) to simulate job lifecycle.
- **Filesystem** – Use `memfs` or Jest's manual mocks for `fs` operations.

## CI Integration
Add a GitHub Actions workflow step:
```yaml
- name: Run tests with coverage
  run: npm test -- --coverage
- name: Enforce coverage
  run: npx jest --coverage --coverageThreshold='{"global": {"branches": 90, "functions": 90, "lines": 90, "statements": 90}}'
```
The workflow will fail the PR if coverage is insufficient.

## Success Criteria
- All `src/lib/` files have associated test files.
- Overall statement coverage ≥ 90 %.
- CI pipeline passes the coverage gate.
- No new test failures introduced.
